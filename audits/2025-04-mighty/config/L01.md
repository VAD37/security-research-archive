# LendingPool no rounding direction when mint/redeem share cause 1wei drifting

## Description

LendingPool's deposit and redeem functions compute eToken (share) conversions using simple integer division:

```solidity
// Deposit conversion (rounds toward zero)
eTokenAmount = amount.mul(exchangeRate).div(Precision.FACTOR1E18);
// Redeem conversion (rounds toward zero)
underlyingTokenAmount = reserve.eTokenToReserveExchangeRate().mul(eTokenAmount).div(Precision.FACTOR1E18);
```

This deviates from the ERC4626 standard (e.g., Solmate), which applies rounding down on mint (deposit) and rounding up on redeem, potentially causing up to 1 wei drift per conversion.

## Recommendation

Align the conversion logic with ERC4626 rounding conventions:

- Round down when calculating shares to mint (deposit) to avoid over-minting.
- Round up when calculating assets to redeem (redeem) to ensure users receive full value.

Implement precise `mulDiv` with rounding direction control or leverage Solmate's `ERC4626` helper functions (`toShare` and `toAmount`) to eliminate drift and provide fair conversions.
